UX Reference — Backend Migration

WebGPU Compute Filter Migration

This document records the UX surface of the WebGPU compute shader filter migration. All migrated filter dialogs and menus are visually and behaviourally unchanged from the user's perspective. The only new UX surface introduced is the WebGPU dispatch error state. Existing design files for each filter dialog remain the authoritative design reference.

Observable UX differences after migration
↑
Faster filter execution (large canvases, high radii).
GPU compute parallelism eliminates the per-pixel sequential cost of WASM execution. Lens Blur at radius 100 on a 4000 × 3000 px canvas — previously multi-second on CPU — should complete in milliseconds on the GPU. The Apply button and canvas preview both respond faster. No spinner, progress bar, or loading state is added; the filter completes fast enough that none is warranted.
↑
Debounced canvas preview is more responsive.
Each parameter adjustment in a filter dialog triggers a debounced preview pass. With compute shaders, preview latency drops significantly for blur filters at high radii, making interactive parameter tuning feel instantaneous rather than sluggish.
–
No WASM init delay on first filter use.
Previously the WASM module needed to be instantiated before the first filter could run, adding a small one-time delay (typically <200 ms). With the WASM paths removed, that delay no longer occurs. Users opening a filter dialog for the first time in a session will see it respond immediately.
–
Pixel output is identical within 1-ULP tolerance.
GPU and CPU floating-point rounding may differ by at most one 8-bit level per channel. This difference is below the threshold of human perception. Users will not notice any change in filter output quality.
!
New failure mode: WebGPU compute dispatch error.
With WASM fallback removed, a GPU dispatch failure now surfaces an error to the user rather than silently falling back to CPU. This failure mode is expected to be rare (driver crashes, out-of-memory on very large dispatches) but must be handled visibly. See Section 3 below for the error state design.
Error state: WebGPU compute dispatch failure
Requirement (from spec): If a WebGPU compute dispatch fails at runtime, the operation must surface an error to the user. It must not silently fall back to a CPU/WASM path or silently no-op. The layer must remain in its pre-filter state (the operation is rolled back / never committed).

After a dispatch failure:
• The error banner appears below the parameter controls.
• The layer remains unmodified (operation was rolled back).
• "Apply" relabels to "Retry" — the user may adjust parameters and try again.
• Closing the dialog with "Close" discards the panel; no undo entry is written.
• Parameter controls remain editable so the user can reduce Radius before retrying.

Instant-apply filters (Sharpen, Sharpen More) have no dialog to inject an error banner into. On dispatch failure an error toast appears at the bottom-right corner of the application window.

Toast behaviour:
• Appears immediately after the failed GPU dispatch.
• Filter name ("Sharpen" / "Sharpen More") appears in the title for clarity.
• The layer is unchanged — no undo entry is written.
• Toast auto-dismisses after 6 s, or on user click of the × button.
• No retry is offered — the user may re-select the menu item to try again.

Copy guidance: Error messages must confirm (a) what was attempted, (b) that the layer was not modified, and (c) a brief actionable hint where one exists (e.g. "reduce Radius" for large-kernel blurs). Do not expose raw WebGPU error codes or WGSL compile errors in the user-facing message — log those to the developer console only.
Out-of-scope filters — WASM retained

The following operations remain in C++/WASM and are not affected by this migration. Their dialogs and behaviour are unchanged and do not gain the error-toast failure mode.

Filter / Operation Reason retained in WASM
Floyd-Steinberg Dithering Sequential error-diffusion pixel dependency — not GPU-parallelisable without algorithmic restructuring.
Flood Fill Branching data-dependent memory access (scanline) maps poorly to GPU compute.
Colour Quantization Palette generation (median cut / k-means) is iterative and CPU-friendly.
Resize Handled separately from the filter pipeline; not in scope for this migration.
Curves / Histogram Already migrated to the WebGPU adjustment compute pass; no second migration needed.
Remove Motion Blur Iterative Richardson-Lucy deconvolution — different parallelisation characteristics.
